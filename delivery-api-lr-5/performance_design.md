# Performance Design: Caching & Rate Limiting Strategy

## 1. Анализ производительности (Performance Analysis)

В данном разделе проводится анализ исходного состояния API сервиса `delivery-api` (Вариант 24: Сервис доставки посылок) для выявления "узких мест" и определения требований к оптимизации.

### 1.1. Определение "горячих путей" (Hot Paths)
На основе анализа бизнес-логики и типичных сценариев использования аналогичных систем (СДЭК), можно выделить следующие операции с высокой частотой вызовов:

1.  **Поиск объектов доставки (`GET /properties`)**:
    *   **Параметры:** Фильтрация по городу (`city`), диапазону цен (`min_price`, `max_price`).
    *   **Частота:** Высокая. Это основная операция для анонимных пользователей и посетителей сайта.
    *   **Характер нагрузки:** Read-heavy (чтение преобладает над записью). Данные меняются редко относительно частоты чтения.
2.  **Поиск пользователей (`GET /users`)**:
    *   **Параметры:** По логину или маске имени/фамилии.
    *   **Частота:** Средняя. Используется при проверке уникальности логина, административных задачах или внутренних сервисах.
3.  **Аутентификация (`POST /login`)**:
    *   **Частота:** Высокая в моменты пиковой активности (утро/вечер), но обычно ниже, чем поиск доставки.
    *   **Риск:** Подвержена brute-force атакам.

### 1.2. Определение медленных операций (Bottlenecks)
Операции, которые могут вызывать задержки из-за обращения к базам данных или сложных вычислений:

1.  **Сложные запросы к MongoDB (`GET /properties`)**:
    *   Несмотря на наличие индексов в MongoDB, запросы с диапазоном цен и текстовым поиском по городу требуют сканирования индексного дерева. При большом объеме данных (миллионы объявлений) это может занимать десятки миллисекунд.
    *   Десериализация BSON документов в C++ объекты также потребляет CPU.
2.  **Запросы к PostgreSQL (`GET /users`)**:
    *   Поиск по маске имени/фамилии (`LIKE '%...%'`) сложен и ресурсозатратен при его использовании большим количеством пользователей.

### 1.3. Требования к производительности
Для обеспечения комфортного пользовательского опыта (UX) и устойчивости системы устанавливаются следующие SLA (Service Level Agreements):

*   **Время отклика (Latency):**
    *   P95 для `GET /properties`: < 100 мс.
    *   P95 для `GET /users`: < 50 мс.
    *   P95 для `POST /login`: < 200 мс (учитывая хеширование пароля).
*   **Пропускная способность (Throughput):**
    *   Система должна выдерживать минимум 1000 RPS (Requests Per Second) на чтение свойств доставки без деградации времени отклика.
    *   Минимум 100 RPS на аутентификацию.
*   **Доступность:** 99.9% во время пиковых нагрузок.

## 2. Проектирование стратегии кеширования (Caching Strategy Design)

Кеширование применяется для снижения нагрузки на базы данных (PostgreSQL и MongoDB) и уменьшения времени отклика для часто читаемых данных.

### 2.1. Выбор данных для кеширования

| Данные | Обоснование | Тип данных |
| :--- | :--- | :--- |
| **Списки объектов доставки** (`GET /properties?city=...`, `GET /properties?min_price=...`) | Часто запрашиваются, данные меняются редко (добавление/изменение объявления происходит реже, чем просмотр). Результаты сложных выборок. | Read-Heavy, Rarely Updated |
| **Профили пользователей** (`GET /users?login=...`) | Данные пользователя почти статичны после регистрации. Используются при каждом запросе, требующем авторизации (если кеш интегрирован в middleware). | Read-Heavy, Static |

### 2.2. Стратегии кеширования

Для реализации будет использоваться **Redis** как внешнее хранилище кеша, интегрированное в C++ сервис через `userver` компоненты. Для поиска доставки (`GET /properties`) и поиска пользователей (`GET /users`) используется стратегия **Cache-Aside (Lazy Loading)**.

#### Пример: поиск доставки

1.  **Чтение:**
    *   Приложение получает запрос, например, `GET /properties?city=Moscow&from=1&to=50`.
    *   Формируется уникальный ключ кеша: `search:{city:Moscow}:1:50`.
    *   Проверка наличия ключа в Redis.
    *   **Hit:** Возврат данных из Redis.
    *   **Miss:** Запрос к MongoDB -> Получение данных -> Запись в Redis с TTL -> Возврат данных клиенту.
2.  **Запись/Обновление:**
    *   При добавлении нового объекта (`POST /properties`):
    *   Приложение явно инвалидирует (удаляет) ключи кеша, связанные с этим городом.

### 2.3. Параметры кеширования (TTL и Eviction)

*   **TTL (Time To Live):**
    *   Для `properties`: **60 секунд**. Это позволит снизить нагрузку на БД в 90% случаев при пиковом трафике, при этом пользователи увидят актуальные данные с задержкой не более минуты.
    *   Для `users`: **300 секунд (5 минут)**. Данные пользователей меняются очень редко.
*   **Максимальный объем памяти:**
    *   Ограничить память Redis под кеш до 512 МБ - 1 ГБ (в зависимости от доступных ресурсов сервера).

### 2.4. Инвалидация кеша
Реализуется паттерн **Write-Invalidate**:
При успешном выполнении `POST /properties` (создание):
*   Удаляются все ключи кеша, начинающиеся с `search:{city:Moscow}:*`.

Реализуется инвалидация по **TTL**:
Недвижимости, отфильтрованные по диапазону цен, хранятся в кэше до истечения TTL (60 секунд). Такой подход позволяет пользователям увидеть актуальные данные с задержкой не более минуты. Для систем управления доставкаю это допустимо.

## 3. Анализ производительности и мониторинг (Performance Analysis & Monitoring)

Этот раздел описывает, как внедренные механизмы влияют на систему и как измерять их эффективность.

### 3.1. Влияние кеширования на производительность

**Преимущества:**
1.  **Снижение нагрузки на MongoDB:**
    *   Без кеша каждый запрос `GET /properties` вызывает выполнение агрегационного пайплайна или `find()` в MongoDB. При 1000 RPS это создает значительную нагрузку на CPU и I/O диска базы данных.
    *   С кешем (при Hit Rate 80%) нагрузка на БД снижается в 5 раз (до 200 RPS). Это позволяет БД быстрее обрабатывать сложные запросы на запись и редкие запросы на чтение.
2.  **Уменьшение времени отклика (Latency):**
    *   Чтение из Redis занимает ~1 мс (сеть + обработка).
    *   Чтение из MongoDB + десериализация в C++ занимает ~10-50 мс (в зависимости от сложности запроса и размера документа).
    *   **Итог:** Ускорение ответа в 10-50 раз для "горячих" запросов.
3.  **Защита от "Шторма запросов" (Thundering Herd):**
    *   При падении БД или ее перезагрузке, кеш может обслуживать старые данные, позволяя системе оставаться частично работоспособной (Graceful Degradation).

**Риски и недостатки:**
1.  **Устаревание данных:** Пользователи могут видеть объекты, которые уже проданы, в течение времени TTL (до 60 сек). Для доставки это обычно допустимо.
2.  **Сложность инвалидации:** При неправильной инвалидации можно получить несогласованные данные.
3.  **Расход памяти:** Требуется дополнительный сервер/контейнер с Redis.

### 3.2. Влияние Rate Limiting на производительность

**Преимущества:**
1.  **Защита от DDoS и Brute-Force:**
    *   Эндпоинт `POST /login` защищен от перебора паролей. Злоумышленник не сможет сделать 1000 попыток входа в секунду.
    *   Эндпоинты поиска защищены от скриптов, пытающихся выкачать всю базу данных объектов.
2.  **Стабильность системы:**
    *   Отсеивание избыточных запросов на раннем этапе (до обращения к БД) освобождает потоки выполнения в Userver и соединения с БД для легитимных пользователей.
    *   Предотвращение исчерпания ресурсов (CPU, RAM, Connections) из-за аномальной нагрузки.

### 3.3. Метрики для мониторинга производительности

Для оценки эффективности внедренных оптимизаций необходимо собирать следующие метрики (можно экспортировать в Prometheus/Grafana или логировать):

#### А. Метрики Кеширования (Redis)
1.  **Cache Hit Rate (Коэффициент попаданий в кеш):**
    *   Формула: `Hits / (Hits + Misses)`
    *   Целевое значение: > 70-80% для эндпоинта `/properties`.
    *   *Как измерить:* Считать количество запросов, вернувших данные из Redis, и общее количество запросов к эндпоинту.
2.  **Cache Miss Rate:**
    *   Показывает, как часто приходится обращаться к БД. Высокий miss rate может указывать на слишком короткий TTL или слишком разнообразный набор запросов (низкая локальность).
3.  **Average Cache Latency:**
    *   Среднее время получения данных из Redis. Должно быть < 5 мс.
4.  **Memory Usage:**
    *   Объем используемой памяти в Redis. Помогает настроить `maxmemory` и политику вытеснения.

#### Б. Метрики Rate Limiting
1.  **Rate Limited Requests Count:**
    *   Количество запросов, отклоненных с кодом `429 Too Many Requests`.
    *   Резкий рост этой метрики может сигнализировать об атаке или ошибке в клиентском приложении (циклические запросы).
2.  **Rejection Rate by Endpoint:**
    *   Процент отклоненных запросов для каждого эндпоинта. Позволяет точечно настраивать лимиты.

#### В. Общие метрики системы (Application & DB)
1.  **P95/P99 Latency:**
    *   Время отклика 95% и 99% запросов. Сравнение значений *до* и *после* внедрения кеша покажет реальный выигрыш для "хвоста" распределения.
2.  **RPS (Requests Per Second):**
    *   Общая пропускная способность сервиса.
3.  **Database Load:**
    *   CPU Usage MongoDB/PostgreSQL.
    *   Количество активных соединений с БД.
    *   *Ожидание:* Снижение нагрузки на БД пропорционально Cache Hit Rate.

### 3.4. Как измерить эффективность кеширования (Experimental Plan)

Для количественной оценки эффективности предлагается провести нагрузочное тестирование с использованием инструментов (например, `k6` или `pytest` с параллельным запуском):

1.  **Этап 1: Базовая линия (Without Cache)**
    *   Отключить кеш в конфигурации приложения.
    *   Запустить нагрузочный тест: 1000 RPS на `GET /properties?city=Moscow&...` в течение 5 минут.
    *   Зафиксировать:
        *   Среднее время отклика (Avg Latency).
        *   P95 Latency.
        *   Загрузку CPU MongoDB контейнера.
        *   Количество ошибок (если есть).

2.  **Этап 2: С включенным кешем (With Cache)**
    *   Включить кеш (Redis).
    *   Прогреть кеш: сделать несколько тысяч запросов с разными параметрами, чтобы заполнить его.
    *   Запустить тот же нагрузочный тест: 1000 RPS на `GET /properties?city=Moscow&...` в течение 5 минут.
    *   Зафиксировать те же метрики.
    *   Дополнительно зафиксировать `Cache Hit Rate` из метрик Redis (`INFO stats`).

3.  **Анализ результатов:**
    *   Сравнить P95 Latency: ожидается снижение с ~50мс до ~5мс.
    *   Сравнить CPU MongoDB: ожидается снижение на 60-80%.
    *   Рассчитать реальный Hit Rate. Если он низкий (<50%), пересмотреть стратегию ключей или TTL.

### 3.5. Вывод

Внедрение кеширования на базе Redis со стратегией Cache-Aside для эндпоинтов поиска доставки и пользователей позволит значительно снизить нагрузку на базы данных (PostgreSQL и MongoDB) и улучшить время отклика системы, особенно при высоких нагрузках. Rate Limiting обеспечит стабильность сервиса, защищая его от злоупотреблений и атак. Комбинация этих методов является стандартной практикой для высоконагруженных REST API и соответствует требованиям архитектуры современных веб-сервисов.
