# Обоснование проектирования схемы данных (Schema Design)

## 1. Встраивание (Embedding) полей `address` и `details`
**Решение:** Хранить адрес (`address`) и специфические характеристики объекта (`details`) как встроенные поддокументы внутри основного документа доставки.

**Обоснование:**
- **Локальность данных и производительность чтения:** Эти данные запрашиваются практически в 100% случаев при просмотре карточки объекта. Встраивание позволяет получить всю необходимую информацию за один запрос к БД (one round-trip), исключая необходимость в JOIN-операциях на уровне приложения или дополнительных запросов.
- **Логическая целостность:** Адрес и характеристики (например, количество комнат, этаж) не имеют самостоятельной ценности вне контекста самого объекта доставки. Они не требуют отдельного управления идентичностью или конкурентного редактирования разными пользователями.
- **Паттерн:** Используется паттерн **Embedded Document**, который оптимизирует систему для сценариев с преобладанием чтения (read-heavy workload), характерных для каталогов доставки.

## 2. Ссылка (Reference) на `owner_id`
**Решение:** Хранить только идентификатор владельца (`owner_id`) в документе MongoDB, в то время как полные данные пользователя хранятся в PostgreSQL.

**Обоснование:**
- **Гибридная архитектура и целостность:** Поскольку аутентификация и профили пользователей управляются в PostgreSQL, ссылка по ID обеспечивает чистое разделение ответственности (Separation of Concerns).
- **Избежание избыточности данных:** Встраивание данных пользователя (имя, телефон, email) привело бы к дублированию информации в каждом документе квартиры. Любое изменение профиля пользователя требовало бы обновления всех связанных с ним документов доставки, что создало бы проблему согласованности данных и увеличило нагрузку на запись (write amplification).
- **Масштабируемость:** Документы в MongoDB остаются легковесными. Детали владельца подгружаются отдельно только при необходимости (например, на странице "Связаться с владельцем"), что соответствует паттерну **Reference**.

## 3. Денормализация поля `city` на верхний уровень
**Решение:** Вынести поле `city` из вложенного объекта `address` в корень документа.

**Обоснование:**
- **Оптимизация запросов и индексации:** Фильтрация по городу - одна из самых частых операций поиска. Индексы MongoDB работают эффективнее и быстрее на полях верхнего уровня по сравнению с вложенными полями (dot notation).
- **Упрощение агрегаций:** Это позволяет использовать простые стадии `$match` в агрегационных пайплайнах без необходимости использования `$unwind` или сложных путей к вложенным полям, снижая нагрузку на CPU сервера БД.

## 4. Полиморфизм через `type` и `details`
**Решение:** Использовать единую коллекцию для всех типов доставки (Квартиры, Дома, Земля), разделяя их с помощью поля-дискриминатора `type`, а уникальные атрибуты хранить в гибкой структуре `details`.

**Обоснование:**
- **Единая коллекция (Single Collection Pattern):** Избегает сложности управления множеством коллекций для схожих сущностей. Это упрощает стратегию индексирования и реализацию глобального поиска.
- **Проблема разреженных данных:** Объекты доставки имеют сильно отличающиеся наборы атрибутов (например, "этаж" есть у квартир, но нет у земельных участков; "площадь участка" есть у земли, но не у квартир). Использование гибкой структуры `details` решает проблему "разреженных таблиц" SQL, где большинство колонок были бы NULL.
- **Расширяемость:** Новые типы доставки можно добавлять без миграции схемы БД или изменения структуры основного документа, просто определяя новые ключи внутри `details`.
