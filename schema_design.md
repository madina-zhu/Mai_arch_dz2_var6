# Обоснование проектирования схемы данных MongoDB для сервиса доставки (Schema Design)

## 1. Встраивание (Embedding) полей `address` и `details`
**Решение:** Хранить адрес отправителя (`address`) и специфические характеристики посылки (`details`) как встроенные поддокументы внутри основного документа посылки.

**Обоснование:**
- **Локальность данных и производительность чтения:** Эти данные запрашиваются практически в 100% случаев при просмотре карточки посылки. Встраивание позволяет получить всю необходимую информацию за один запрос к БД (one round-trip), исключая необходимость в JOIN-операциях на уровне приложения или дополнительных запросов.
- **Логическая целостность:** Адрес и характеристики (например, габариты, хрупкость) не имеют самостоятельной ценности вне контекста самой посылки. Они не требуют отдельного управления идентичностью.
- **Паттерн:** Используется паттерн **Embedded Document**, который оптимизирует систему для сценариев с преобладанием чтения (read-heavy workload), характерных для трекинга посылок.

## 2. Ссылка (Reference) на `sender_id` и `recipient_id`
**Решение:** Хранить только идентификаторы отправителя (`sender_id`) и получателя (`recipient_id`) в документах MongoDB, в то время как полные данные пользователей хранятся в PostgreSQL.

**Обоснование:**
- **Гибридная архитектура и целостность:** Поскольку аутентификация и профили пользователей управляются в PostgreSQL, ссылка по ID обеспечивает чистое разделение ответственности (Separation of Concerns).
- **Избежание избыточности данных:** Встраивание данных пользователя привело бы к дублированию информации в каждом документе посылки. Любое изменение профиля пользователя требовало бы обновления всех связанных с ним документов, что создало бы проблему согласованности данных.
- **Масштабируемость:** Документы в MongoDB остаются легковесными. Детали отправителя/получателя подгружаются отдельно только при необходимости.

## 3. Денормализация полей `from_city` и `to_city` на верхний уровень
**Решение:** Вынести поля `from_city` и `to_city` из вложенного объекта `address` в корень документа.

**Обоснование:**
- **Оптимизация запросов и индексации:** Фильтрация по городу отправления или назначения - одна из самых частых операций поиска. Индексы MongoDB работают эффективнее на полях верхнего уровня.
- **Упрощение агрегаций:** Это позволяет использовать простые стадии `$match` в агрегационных пайплайнах без использования `$unwind` или сложных путей к вложенным полям.

## 4. Полиморфизм через `type` и `details`
**Решение:** Использовать единую коллекцию для всех типов посылок (обычная, хрупкая, крупногабаритная, экспресс), разделяя их с помощью поля-дискриминатора `type`, а уникальные атрибуты хранить в гибкой структуре `details`.

**Обоснование:**
- **Единая коллекция (Single Collection Pattern):** Избегает сложности управления множеством коллекций для схожих сущностей. Это упрощает стратегию индексирования и реализацию глобального поиска.
- **Проблема разреженных данных:** Разные типы посылок имеют сильно отличающиеся наборы атрибутов (например, "max_weight" есть у экспресс-доставки, но нет у обычной; "is_fragile" есть у хрупких, но не у обычных). Использование гибкой структуры `details` решает эту проблему.
- **Расширяемость:** Новые типы посылок можно добавлять без миграции схемы БД.
